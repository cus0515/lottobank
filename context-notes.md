# LottoBank 연금 판매소 API 연동 컨텍스트 노트

- 공식 연금 결과 페이지의 당첨판매점 버튼은 `/wnprchsplcsrch/home?ltGds=pt720&ltEpsd=회차`로 이동한다.
- 해당 판매소 페이지 내부에서 연금복권은 `/wnprchsplcsrch/selectPtWnShp.do`를 호출한다.
- 기존 후보였던 `/pt720/selectPstPt720WnShpList.do`와 `/pt720/selectPstPt720WnShpInfo.do`는 JSON이 아니라 HTML을 반환한다.
- 321회 기준 `/wnprchsplcsrch/selectPtWnShp.do?srchWnShpRnk=1&srchLtEpsd=321&srchShpLctn=`은 1건, `srchWnShpRnk=2`는 4건을 반환한다.

- `functions/api/lotto-region.js`의 연금 분기는 `selectPtWnShp.do` 단일 호출로 변경하고, `wnShpRnk` 값이 `1`, `2`인 항목만 각각 `stores1`, `stores2`로 분리한다.
- 연금 판매소 응답에는 자동/수동 필드가 없으므로, UI 요약은 자동/수동 값이 있을 때만 표시하고 없으면 `판매소 N`으로 표시한다.
- 직접 함수 호출 검증 결과 321회 연금은 1등 1곳, 2등 4곳을 반환했고, 1230회 로또 판매소도 정상 응답했다.
- `index.html`의 실행 가능한 인라인 스크립트 3개는 `vm.Script` 문법 검사를 통과했다. JSON-LD 스크립트는 실행 스크립트가 아니어서 검사 대상에서 제외했다.

## 홈 및 실시간 인증 피드 개편

- 기존 `home` 화면에 최신 당첨번호, 최근 회차 결과, 실시간 활동 피드가 이미 있으므로 이 화면을 정식 홈페이지로 사용한다.
- 기존 `fortune` 경로는 저장된 링크와 호출부의 호환성을 위해 유지하되, 화면 내용은 번호 분석 대신 실시간 당첨 인증 피드만 렌더링한다.
- 모바일 하단 메뉴는 홈, 회차조회, QR 인증, 전적, 인증피드, 더보기 순으로 구성한다. 뱃지는 더보기 메뉴에서 계속 접근할 수 있다.
- 연금복권 등수별 당첨 현황은 로또와 같은 `prize-table` 스타일을 사용해 글꼴 크기, 셀 여백, 정렬을 통일한다.
- 대시보드의 세 번째 안내 항목은 번호 분석 대신 실시간 당첨 인증으로 연결한다.
- 배포 화면을 1440×900과 390×844에서 확인했으며 홈과 인증 피드에 가로 넘침이 없었다.
- 연금 당첨 현황 표는 모바일에서 마지막 열이 밀리지 않도록 고정 열 비율과 셀 내부 줄바꿈을 적용한다.

## 대시보드, 실시간 피드, 랭킹 및 채팅 개편

- `home` 경로는 최신 당첨정보 대신 실시간 활동 피드 전용 화면으로 사용한다.
- 별도 `fortune` 메뉴는 제거하고 이전 경로 호출은 `home`으로 이동시켜 기존 링크 오류를 막는다.
- `판매왕`은 현재 판매자 데이터가 없으므로 QR 인증량이 많은 사용자 카테고리로 정의한다.
- 회차별 1등과 2등 칭호는 `[1230회차 1등] 닉네임` 형태로 랭킹과 채팅에 동일하게 표시한다.
- 채팅 닉네임은 기존처럼 `profiles.nickname`을 우선 사용하고, `verified_title`을 닉네임 앞에 결합한다.
- 배포 화면에서 로또와 연금 카운트다운이 초 단위로 표시되는 것을 확인했다.
- 연금 판매소는 판매점과 주소를 왼쪽, 구분을 가운데로 정렬하고 지역 버튼의 필터 상태와 행 개수가 정상 변경되는 것을 확인했다.
- 데스크톱 내비게이션은 홈, 회차 조회, 랭킹만 표시하며 홈에는 실시간 피드만 표시된다.

## 헤더와 결과 화면 시각 조정

- 상단 브랜드가 작게 보인 원인은 `.logo-title span` 규칙이 브랜드 글자까지 11px로 덮어쓴 것이므로 브랜드와 설명 문구 클래스를 분리한다.
- 좌측 패널은 막대그래프 아이콘과 `LottoBank`만 표시해 서비스 고유 대시보드 영역으로 구분한다.
- 회차 헤더는 그라데이션을 제거하고 로또는 `var(--lotto-red)`, 연금은 `PENSION_GREEN`과 같은 색 계열을 사용한다.
- 테마 전환은 텍스트보다 해와 달의 선형 SVG 아이콘을 사용해 헤더 공간과 의미 전달을 유지한다.
- 배포 화면에서 브랜드는 데스크톱 22px, 모바일 20px로 표시되며 가로 넘침이 없음을 확인했다.
- 로또 회차 헤더는 `rgb(225, 29, 72)`, 연금 회차 헤더는 `rgb(22, 163, 74)`로 활성 탭 색상과 일치한다.
- 테마 버튼 클릭 시 다크 테마와 해 아이콘, `라이트 모드로 전환` 접근성 레이블로 정상 변경된다.

## 결과 회차 헤더와 당첨표 통일

- 회차 헤더는 `var(--paper-white)`와 `var(--text-dark)`를 사용해 라이트모드에서는 흰색, 다크모드에서는 어두운 패널색으로 자동 대응한다.
- 회차 숫자는 로또 빨강, 연금 초록으로 복권 종류만 구분한다.
- 두 당첨표의 열은 `등수`, `조건`, `당첨수`, `당첨금`, `구입 형태` 순서로 통일한다.
- 연금의 지급 방식 값은 조건 열에, 기존 총액 값은 당첨금 열에 유지한다.
- 배포 화면에서 회차 헤더는 라이트모드 `rgb(255, 255, 255)`, 다크모드 `rgb(30, 41, 59)`로 표시된다.
- 로또와 연금 표 모두 `등수`, `조건`, `당첨수`, `당첨금`, `구입 형태` 순서로 표시됨을 확인했다.
- 모바일 390px에서 연금 표가 패널 너비 안에 표시되고 가로 넘침이 없음을 확인했다.

## 랭킹, 명예의 전당, 지도 개편

- 홈 패널 제목이 이미 실시간 피드이므로 내부 카드 제목은 제거한다.
- 월간 랭킹은 구매, 저격, 수익, 똥손, 개척으로 구성하고 매월 결과를 스냅샷으로 보관한다.
- 누적 랭킹은 기부왕, 출석왕, 개근왕, 3·4·5등 컬렉터, 스캔왕, 자동왕, 수동왕으로 구성한다.
- 저격과 똥손에는 게임별 적중 개수가 필요하고 판매점 정보는 QR 등록 시 티켓에 함께 저장한다.
- 기부왕은 별도 기부 데이터가 아니라 `누적 구매금액 - 누적 당첨금액`이 큰 사용자로 계산한다.
- 출석왕은 로그인 사용자의 누적 접속일 수, 개근왕은 최장 연속 접속일 수로 집계하며 하루 첫 접속만 기록한다.
- 배포 URL에서 데스크톱·모바일 랭킹과 모바일 지도 화면을 확인했으며 가로 넘침이 없었다.
- 명예의 전당은 티켓의 복권 종류, 회차, 당첨 등수와 사용자 프로필을 기준으로 영구 보관한다.
- 월간 칭호는 사용자가 선택하지 않고 시스템 집계 결과만 채팅에 노출한다.
- 지도는 OpenStreetMap으로 현재 위치를 표시하고 카카오 지도 주변 로또 판매점 검색으로 연결한다.

## 랭킹 헤더와 누적 아이콘 정리

- 별도 LEADERBOARD 배너는 제거하고 랭킹·명예의 전당 전환 탭을 패널 헤더에 배치한다.
- 누적 항목 아이콘은 기부 💝, 출석 🗓️, 개근 🔥, 3등 🥉, 4등 4️⃣, 5등 5️⃣, 스캔 📸, 자동 🤖, 수동 ✍️로 구분한다.
- 배포 URL의 데스크톱과 390px 모바일에서 헤더 탭, 배너 제거, 누적 아이콘 및 가로 넘침 없음을 확인했다.

## 누적 랭킹과 별도 관리자 복구

- 누적 랭킹은 기부왕, 출석왕, 개근왕, 스캔왕, 자동왕, 수동왕, 3·4·5등 콜렉터, 대개척자 순서로 표시한다.
- 대개척자는 QR 티켓에 저장된 누적 고유 판매점 수를 기준으로 집계한다.
- 별도 관리자 페이지는 허용된 티켓과 전적 데이터만 수동 입력하며 삭제 시 테이블별 실제 기본키를 사용한다.
- 배포에서 메뉴 순서, 누적 랭킹 10개 순서, 별도 관리자 로그인·직접 입력·삭제 호출 구조와 모바일 가로 넘침 없음을 확인했다.
- 운영 데이터의 실제 입력·삭제는 수행하지 않았으며 Supabase RLS가 관리자 쓰기를 허용해야 저장과 삭제가 성공한다.

## 지도, 로그인 메뉴, QR 인증 개편

- OSM 임베드의 기부 문구를 제거하기 위해 Leaflet과 표준 OSM 타일을 사용하며 라이선스 출처 표시는 유지한다.
- 주변 판매점 목록은 현재 위치 기준 1km 이내만 거리순으로 표시한다.
- 예시 QR에서 회차 1222, 5게임 번호와 후미 식별값을 파싱할 수 있고 구매일·판매점·자동/수동은 추출할 수 없다.
- 오프라인 구매는 판매점명과 주소를 직접 입력하고 인터넷 구매는 판매점을 동행복권 인터넷 구매로 기록한다.
- QR 등록 시 현재 위치는 사용하지 않고 기존 판매점을 검색해 선택하거나 판매점명과 주소를 직접 등록한다.
- 별도 admin.html은 유지하고 메인 SPA 내부 관리자 진입점과 페이지는 제거한다.
- 배포에서 Leaflet 타일 6장 렌더링, 기부 문구 제거, 1km 목록 영역, 게스트 메뉴 숨김과 390px 가로 넘침 없음을 확인했다.

## 연금 판매점과 관리자 전체 검증

- 배포 API는 321회 연금 판매점 1등 1건과 2등 4건을 반환하지만 외부 API 장애 시에도 같은 공식 데이터를 폴백한다.
- 관리자 검증 데이터는 관리자 본인 계정에만 생성하고 `admin-test:` 식별값으로 일괄 정리한다.
- 관리자 전체 권한은 `jobdevil23@gmail.com` JWT에만 적용하며 일반 사용자 RLS는 유지한다.

## QR 등록과 통합 데이터베이스

- 오프라인 판매지점은 공개 판매점 목록을 직접 수정하지 않고 사용자별 `user_lottery_stores`에 저장한다.
- QR 등록 시 선택한 개인 지점의 이름과 주소를 티켓 스냅샷으로 남겨 이후 지점명이 바뀌어도 구매 이력을 보존한다.
- 카메라 인식 직후 자동 인증하지 않고 URL 입력과 미리보기까지만 수행하여 구매 방식과 개인 지점을 먼저 확인하게 한다.
- 티켓 저장이 실패하면 로컬 중복 키를 기록하지 않아 사용자가 같은 QR로 다시 시도할 수 있게 한다.
- `database_setup_v2.sql`은 현재 UI에서 사용하는 테이블과 관리자 검증용 전체 CRUD 정책을 한 번에 구성한다.
- 배포 화면에서 개인 판매지점 선택과 모바일 QR 레이아웃을 확인했고, 연금 321회 API는 1등 1건과 2등 4건을 반환한다.
## QR 화면과 대시보드 정리

- 로그인 전용 탭은 상시 연두색을 사용하지 않고 일반 탭과 같은 기본색과 선택색을 사용한다.
- 판매지점 관리는 QR 인증에서 분리하고 추후 나의 전적 화면에서 구현한다.
- 대시보드 활동 피드는 로그인 여부와 관계없이 제거하며 실시간 피드 본 화면은 유지한다.
- 비로그인 안내는 QR 인증, 전적 자동 기록, 랭킹 반영의 현재 서비스 흐름을 설명한다.
- 배포 화면에서 판매지점과 대시보드 활동 피드 제거, 안내 문구와 QR 감지 코드 반영을 확인했다.
## 관리자 전체 검증과 QR 2단계 등록

- QR URL에서는 회차, 게임 번호, 게임 수와 고유값만 확정할 수 있다.
- 자동·수동 여부와 발행일은 QR URL에 없으므로 QR 등록 화면에서 제거한다.
- QR 고유값은 `tickets.qr_url`의 유일키로 저장하지만 화면에는 노출하지 않는다.
- QR 흐름은 카메라 또는 URL 조회, 복권 용지형 미리보기, 등록 순서로 분리한다.
- 관리자 페이지는 핵심 테이블을 공통 JSON 편집기로 직접 검증할 수 있게 확장한다.
- 원격 Supabase 핵심 테이블 조회가 모두 404로 확인되어 통합 SQL 실행 전에는 실제 QR 저장 검증이 불가능하다.
- 관리자 전체 데이터 탭의 DB 연결 점검으로 스키마 적용 여부를 한 번에 확인할 수 있다.
## 채팅 뱃지와 당첨 칭호 스타일

- 관리자 시나리오 칭호와 실제 채팅 칭호를 `로또 N등`, `연금 N등`, `로또+연금 N등` 형식으로 통일한다.
- 채팅 전송 직전에 시스템 칭호를 다시 조회하여 관리자에서 만든 검증 데이터가 즉시 반영되게 한다.
- 2등 스타일은 로또, 연금, 동시 당첨 모두 실버 계열이며 네온과 애니메이션을 사용하지 않는다.
- 관리자 뱃지 풀세트는 중복 뱃지가 있어도 전체 저장이 실패하지 않도록 복합키 upsert를 사용한다.
- 기능 검증은 시나리오 생성 후 DB 사후조건, 반복 생성·정리, 두 창 실시간 반영, 반응형 시각 회귀 순서로 확장할 수 있다.
- 배포 화면에서 채팅 간격 14px, 닉네임 콜론 제거, 동시 1등 그라데이션과 2등 네온 제거를 확인했다.
## 당첨 뱃지 색상 재조정

- 로또와 연금 단독 1등은 각각 빨강과 초록을 유지하되 채도를 낮춰 함께 표시될 때 충돌을 줄인다.
- 로또·연금 동시 1등은 은색 단색 바탕과 은빛 네온을 사용한다.
- 로또와 연금 단독 2등은 평면 은색을 사용하고, 로또·연금 결합 2등은 은색 네온을 사용한다.
- 배포 화면에서 로또·연금 1등과 2등의 은색 네온, 단독 2등의 평면 은색, 가운데점 표기를 확인했다.
## 결합 1등과 2등 색상 개선

- 로또·연금 1등은 빨강과 연두 사이에 밝고 중립적인 연결색을 넣어 경계가 탁해지지 않게 한다.
- 2등은 회색 대신 채도가 낮은 아이스 블루를 사용하고 네온 없이 1등보다 약하게 표시한다.
- 배포 HTML에서 새 빨강·연두 그라데이션과 아이스 블루 2등 스타일 반영을 확인했다.
## 전적과 인증 표시 통합

- 전적 분류는 집계 테이블이 아니라 `tickets.lottery_type`을 원천으로 계산한다.
- 명예의 전당과 피드 닉네임은 `profiles.nickname`을 우선 사용한다.
- 프로필에서는 당첨 칭호와 뱃지를 노출하지 않고 채팅 칭호에서만 표현한다.
- 변경되는 피드와 명예의 전당 아이콘은 기존 `SVG_ICONS`를 재사용한다.

## 복합 당첨 타이틀 분리 표시

- 저장된 `로또·연금 N등` 값은 변경하지 않고 채팅 렌더링만 두 배지로 분리한다.
- 1등은 빨강과 연두를 연결하고, 2등은 강조도를 낮춘 은회색 네온을 사용한다.

## 월간 랭킹 뱃지와 QR 번호 이력

- 월간 랭킹은 `dense_rank`로 공동 1위를 허용하고 티켓 변경 및 로그인 시 재집계한다.
- 채팅에는 당첨 타이틀, 월간 1위 뱃지, 닉네임 순서로 표시한다.
- 연금 QR의 미확인 중간값은 의미를 추정하지 않고 `serial_meta` 원문으로 보존한다.
- 로또는 전체 게임 번호를 `game_numbers`, 연금은 조와 6자리 번호를 별도 필드에 저장한다.
- 복합 1등은 빨강·초록 단일 연결 프레임, 복합 2등은 한 단계 약한 연보라 연결 프레임으로 표시한다.

## QR·전적 상단 탭과 관리 UI 정리

- QR과 전적의 분류 탭은 중앙 패널 헤더에서 전환한다.

## 전적·랭킹·모바일 통합 보완

- 복권 종류는 과거 데이터 충돌을 고려해 `result='pension'`을 우선 판정한다.
- 다중 로또 게임은 `game_numbers`가 없으면 QR 원문을 다시 파싱해 복원한다.

- 인터넷 구매는 공식 계정 자동 수집 API가 없어 동행복권 구매내역의 QR 주소 등록 방식으로 통합한다.
- 파일 감사 결과 현재 추적 파일은 배포·관리·DB·캐시·자동 갱신 중 하나에 사용되므로 이번 작업에서 삭제하지 않는다.
## 판매점 선택과 운영 보완

- 판매점은 사용자가 자유 입력하지 않고 `lottery_stores`의 승인 목록에서 검색해 게임별로 지정한다.
- 게임별 선택 결과는 기존 `ticket_game_stores`에 저장하고, 선택창은 별도 팝업으로 제공한다.
- 공식 전체 판매점 공개 API는 확인 전까지 가정하지 않으며 승인 목록이 비어 있으면 운영 데이터 등록 안내를 표시한다.
- 전적 화면은 중간 패널 자체를 외곽 컨테이너로 사용하고 추가 외곽 카드 표현은 만들지 않는다.
- 단독 로또 1등은 금색, 단독 로또 2등은 은색으로 맞추고 동시 당첨은 이중 테두리 효과로 구분한다.
- 회원가입 제출 직전에 닉네임을 다시 조회해 비동기 입력 검사와 제출 사이의 중복 생성을 막는다.
- 연금 판매소 응답은 `data.list`, `data.contents`, 최상위 `list`를 모두 지원하고 등수 문자열에서 숫자만 추출한다.
- 브라우저 측정에서 데스크톱 세 패널 헤더는 모두 50px이며 모바일 390px 화면의 가로 넘침은 0px로 확인했다.
- 단순 정적 서버에서는 Cloudflare Functions가 실행되지 않아 발생하는 API JSON 경고만 있었고 인라인 스크립트와 판매소 Worker 모듈 문법은 통과했다.

## 클로드 업데이트 복구 결정

## 전체 기간 랭킹 집계 기준
- 기부왕은 누적 구매금액에서 누적 당첨금을 뺀 금액을 사용한다.
- 출석왕은 중복 없는 접속일 수, 개근왕은 최장 연속 접속일 수를 사용한다.
- 스캔왕은 중복 방지된 QR 티켓 수를 사용한다.
- 자동왕과 수동왕은 `purchase_mode`가 확인된 티켓의 게임 수만 집계한다.
- 3·4·5등 콜렉터는 `game_results`의 게임별 결과를 우선 사용하고 구형 데이터는 티켓 대표 등수로 보완한다.
- 월간 저격은 비율이 아니라 월간 적중 번호 총개수를 기준으로 유지한다.

## 인터넷 구매 캡처 인증 기준
- 캡처 이미지는 브라우저 안에서 OCR하고 원본 이미지는 서버에 저장하지 않는다.
- 티켓 하단 30자리 고유 코드를 공백 없이 정규화해 `ticket_code`로 저장한다.
- QR URL의 시리얼과 인터넷 캡처의 고유 코드는 같은 `ticket_code` 중복 기준을 사용한다.
- 인터넷 구매는 `purchase_channel='online'`, 판매점은 `동행복권 인터넷 구매`로 기록한다.
- OCR 결과는 등록 전에 티켓 미리보기로 사용자가 확인한다.
- 제공된 1231회 샘플에서 5게임 번호, 자동 구매, 구매일, 30자리 고유 코드를 정확히 추출했다.
- 추첨 전 구매는 `pending`으로 저장하며 추첨번호가 제공된 뒤 로그인 시 결과를 갱신한다.
- 추첨 대기 티켓은 월간 저격과 똥손 점수에서 제외한다.

## 인터넷 구매 판매점
- 인터넷 구매 판매점은 공용 고정 ID `dhlottery_online`을 사용한다.
- 티켓의 `store_id`와 게임별 `ticket_game_stores`를 함께 저장한다.
- 인터넷 티켓의 모든 게임은 `동행복권 인터넷 구매`로 자동 지정한다.
- 연금복권 인터넷 캡처는 발행 시각, 회차, 1~5조 번호 조합을 중복 식별값으로 사용한다.
- OCR에서 조 글자가 깨질 수 있어 티켓에 표시된 다섯 번호 행의 순서를 1~5조로 확정한다.

## 관리자 운영 권한
- 관리자 여부는 이메일 문자열이 아니라 `app_admins.user_id`로 판정한다.
- 전체 회원 초기화 후 첫 관리자는 SQL Editor에서 이메일로 한 번 지정한다.
- 관리자 페이지 경로를 숨기는 것은 보조 수단이며 실제 권한은 DB 함수에서 재검사한다.
- 데이터 초기화는 계정 유지 초기화와 인증 회원 포함 전체 초기화를 분리한다.
- 관리자 페이지 기본 경로는 `/lotto-control-8f3c2a71.html`이며 파일명을 바꿔 재배포할 수 있다.
- AdSense는 승인 후 중앙 콘텐츠 하단과 피드·회차 조회 사이에 배치하고 QR·로그인·채팅 입력 주변은 제외한다.

## 회원가입 인증
- 가입 화면의 세로 문구 문제는 체크박스가 일반 입력창의 `width:100%`를 상속한 것이 원인이었다.
- 로그인은 비밀번호 방식을 유지하고 가입 확인만 6자리 이메일 OTP로 바꾸는 편이 사용자 부담과 보안 균형이 좋다.
- 발신자 신뢰도와 스팸 분류 개선은 Supabase 기본 메일이 아니라 전용 SMTP, SPF, DKIM, DMARC 설정이 필요하다.

## 1231회 결과 누락
- 공식 새 API의 1231회 당첨번호는 기존 화면과 일치했지만 GitHub 캐시 생성기가 1등 정보까지만 저장했다.
- API가 GitHub 캐시를 최우선 반환하여 2~5등 당첨자 수와 총 판매액이 0 또는 미집계로 표시됐다.
- 캐시는 2~5등 당첨자 수와 총 판매액이 모두 있을 때만 완전한 결과로 사용한다.
- 배포 API에서 1231회 2~5등 인원과 총 판매액이 공식 값으로 반환되는 것을 확인했다.

## 관리자 데이터 관리 정리
- 운영 상태 카드는 단순 집계 표시가 아니라 해당 테이블의 데이터 관리 화면으로 이동하는 진입점으로 사용한다.
- 전체 데이터 화면에서 행을 선택하면 기본키 조건과 현재 JSON을 자동 입력해 초보자도 수정·삭제 대상을 식별할 수 있게 한다.
- 비로그인 통계와 빈 피드는 현재 DB 상태를 반영하므로 가짜 데이터로 취급하지 않는다.
- 공개 SPA에 남아 있던 구 관리자 패널, 샘플 전적 렌더러, 관리자 검증용 Mock 분기는 별도 관리자 페이지와 중복되어 제거했다.
# 번호 레이더와 랭킹 연동 보완

- 레이더의 1차 탭은 복권 종류를 구분하고 추천 이력은 화면 내부 필터로 제공한다.
- 연금 레이더는 회차 결과를 반복 노출하지 않고 추천 번호와 자리별 출현 분석에 집중한다.
- 회원 수는 전적 생성 여부와 무관하도록 `profiles` 행 수를 기준으로 집계한다.
- 출석 저장 직후 `refresh_all_time_rankings`를 직접 호출해 트리거 설치 상태에 따른 갱신 누락을 줄인다.
- 추천 이력은 기존 `saved_number_sets`에 `source='radar'`로 저장해 직접 저장 번호와 분리한다.
- 추천 결과는 적중 개수가 아니라 로또 1~5등과 연금 1~7등·보너스 규칙으로 표시한다.
- 사용자 요청 범위를 벗어나는 QR과 전적 등록 흐름은 변경하지 않는다.

# DB 전체 연동 보완

- 공개 화면에서 회원 수 2명은 정상 조회되며, 티켓 기반 수치가 0인 원인은 티켓 RLS와 빈 집계 스냅샷이다.
- 공개 통계는 개별 티켓을 노출하지 않는 `get_public_platform_stats`, `get_public_number_frequency` RPC로 제공한다.
- `user_stats`는 클라이언트 누적값이 아니라 `tickets` 변경 트리거로 전량 재계산해 삭제·수정 후에도 일치시킨다.
- 기존 회원의 누락된 프로필과 전적 행을 보완한 뒤 월간·전체 기간 랭킹을 즉시 다시 생성한다.

# 전체 UI 규격 통일

- 기존 화면 구조와 색상은 유지하고 마지막 CSS 계층에서 공통 규격을 적용한다.
- 같은 기능의 패널 헤더, 탭, 카드, 보조 문구 크기를 우선 통일한다.
- 작은 화면에서는 카드 수를 줄이지 않고 여백과 열 수를 조절해 가독성을 확보한다.
- 모바일에서는 홈에만 좌측 대시보드를 표시하고 다른 탭은 본문 패널만 표시한다.
- 번호 레이더의 복권 탭은 패널 헤더로 이동하고 연금 레이더는 연두색 계열을 사용한다.
- 실제 DB 검증 결과는 회원 3명, 출석 4건, 출석 랭킹 3행이며 전적 트리거와 추천 이력 구분 열이 설치됐다.

## 전체 UI와 번호 분석 정돈
- 기존 기능과 데이터 로직은 유지하고 마지막 CSS 계층과 레이더 렌더링 마크업만 조정한다.
- 추천 이력 링크에는 사용자가 확인할 수 있는 가장 최근 당첨 결과 회차를 함께 표시한다.
- 분석 영역은 장식 카드보다 지표, 비교값, 추세가 먼저 보이는 증권 앱형 구성을 사용한다.

## 레이더 시각 효과 보완
- 레이더와 연결되는 스캔, 번호 등장, 막대 상승 효과만 적용하고 반복 애니메이션은 추가하지 않는다.
- 화면을 다시 열 때 짧게 한 번 실행하며 모션 감소 환경에서는 모든 추가 효과를 중지한다.
- 배포본에서 로또와 연금 각각 추천 행 5개와 번호 30개가 정상 표시되고 모바일 가로 넘침이 없음을 확인했다.

## 전체 화면 진입 효과 통일
- 공통 페이지 진입 클래스를 탭 전환마다 재적용해 동적으로 렌더링되는 데이터에도 동일한 효과를 제공한다.
- 입력·채팅·긴 본문은 정적인 상태를 유지하고 결과 번호, 지표, 목록처럼 변화 확인이 필요한 요소에만 적용한다.
- 배포본에서 결과 번호, 랭킹 카테고리, 다크모드와 모바일·데스크톱 레이아웃을 검증했으며 가로 넘침이 없다.
# AdSense ads.txt 상태
- AdSense 스크립트는 `ca-pub-3323488098461026`를 사용한다.
- Google `ads.txt` 항목은 `ca-` 접두사 없이 `pub-3323488098461026`로 작성해야 한다.
- Cloudflare Pages 루트에 `ads.txt`가 없어서 AdSense에서 `찾을 수 없음`으로 표시된 상태였다.
- 루트 정적 파일로 `ads.txt`를 추가하면 배포 후 `https://lottobank.pages.dev/ads.txt`에서 확인할 수 있다.

# Search Console 사이트맵 상태
- 네이버 URL 검사는 정상이고 구글 Search Console에서만 사이트맵을 가져올 수 없다고 표시됐다.
- 기존 사이트맵은 단일 SPA 내부 탭을 `#result`, `#rank` 같은 해시 URL로 나눠 제출하고 있었다.
- 사이트맵은 검색에 노출할 canonical 절대 URL 중심으로 제출하는 것이 맞으므로 현재 구조에서는 루트 URL만 남겼다.
- 탭별 검색 노출이 필요하면 해시가 아니라 실제 경로 라우팅과 페이지별 canonical, title, description을 별도로 만들어야 한다.
# 연금복권 번호 이력 복구
- 연금 QR `pd1203201s829103`에는 회차 120, 조 3, 번호 829103이 들어 있다.
- 번호 이력 렌더링은 로또만 `qr_url`에서 번호를 복구했고 연금은 `pension_number` 컬럼만 읽어서 기존 빈 데이터가 `번호 미등록`으로 보였다.
- `restorePensionTicketFromQr`를 추가해 `pension_number`가 비어 있으면 `qr_url`을 다시 파싱해 조와 번호를 채운다.
- 과거 `p` 접두어가 붙은 연금 QR 키도 읽을 수 있도록 연금 보조 포맷을 `p?` 허용으로 넓혔다.

# 레이더 정리와 인터넷 캡처 붙여넣기
- 랭킹 화면의 3등 타이틀 예시는 사용자가 실제 칭호처럼 오해할 수 있어 예시에서 제거한다.
- 레이더 하단은 추천번호 중심으로 단순화하고 당첨번호 요약과 AI 분석 카드는 회차 조회 화면과 역할이 겹쳐 제거한다.
- 추천 적중 이력은 저장된 레이더 추천과 보유한 당첨 결과를 비교해 1~5등 횟수로 표시한다.
- 금액 축약 표기는 서비스 전체에서 금액 감각이 달라 보이므로 원 단위 표기로 통일한다.
- 인터넷 구매 캡처는 파일 업로드와 클립보드 이미지 붙여넣기를 같은 OCR 처리 경로로 보낸다.

# 연금 레이더 적중 이력 보완
- 연금 레이더의 `최근 결과`는 캐시 회차가 아니라 추천 대상 회차의 직전 회차를 기준으로 표시한다.
- 연금복권은 1~7등과 보너스가 있으므로 로또와 같은 5칸 요약을 쓰면 실제 추천 이력과 맞지 않는다.
- 레이더 적중 이력은 추천 이력 화면과 같은 판정 규칙을 공유해 결과가 나온 추천 회차만 누계한다.
