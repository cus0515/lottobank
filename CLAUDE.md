# LottoBank Project Instructions

## 프로젝트 개요

LottoBank는 QR 기반 로또 인증 및 전적 관리 플랫폼이다.

사용자는 로또 QR을 등록하고 자신의 구매 이력, 당첨 이력, 전적, ROI, 뱃지, 랭킹을 관리할 수 있어야 한다.

서비스의 핵심 가치는 실제 구매 데이터와 인증 데이터를 기반으로 한 기록 관리, 통계, 인증, 커뮤니티에 있다.

## QR 시스템 규칙

모든 데이터는 QR 등록을 기준으로 동작한다.

사용자는 번호를 직접 입력하지 않는다.

QR 등록 시 구매 이력이 생성된다.

동일 QR 데이터는 중복 등록될 수 없다.

사용자 전적, ROI, 뱃지, 랭킹은 QR 데이터를 기반으로 자동 계산한다.

## 코딩 행동 지침

### 1. 코딩 전에 먼저 생각한다

가정하지 않는다. 혼란을 숨기지 않는다. 트레이드오프를 명확히 한다.

구현 전에.

- 가정을 명시적으로 밝힌다. 불확실하면 질문한다.
- 여러 해석이 가능하면 제시한다. 임의로 선택하지 않는다.
- 더 단순한 접근이 있으면 말한다. 필요하면 반론을 제기한다.
- 불명확한 것이 있으면 멈춘다. 무엇이 혼란스러운지 명확히 한다.

### 2. 단순함 우선

문제를 해결하는 최소한의 코드만 작성한다. 추측성 코드는 작성하지 않는다.

- 요청받지 않은 기능은 추가하지 않는다.
- 단일 사용 코드에 추상화를 넣지 않는다.
- 요청받지 않은 유연성이나 설정 가능성을 추가하지 않는다.
- 불가능한 시나리오에 대한 에러 처리를 넣지 않는다.
- 200줄을 50줄로 줄일 수 있다면 다시 작성한다.

### 3. 외과적 변경

반드시 필요한 것만 수정한다. 내가 만든 문제만 정리한다.

기존 코드를 수정할 때.

- 인접한 코드, 주석, 포맷을 임의로 개선하지 않는다.
- 동작하는 코드를 리팩토링하지 않는다.
- 내 스타일과 달라도 기존 스타일을 맞춘다.
- 관련 없는 죽은 코드를 발견하면 언급만 한다. 삭제하지 않는다.

내 변경으로 인한 고아 코드는.

- 내 변경이 만든 미사용 import/변수/함수는 제거한다.
- 기존에 있던 죽은 코드는 요청받기 전까지 삭제하지 않는다.

기준. 변경된 모든 줄은 사용자 요청과 직접 연결되어야 한다.

### 4. 목표 중심 실행

성공 기준을 정의한다. 검증될 때까지 반복한다.

작업을 검증 가능한 목표로 변환한다.

- "유효성 검사 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트 작성 후 통과시키기"
- "X 리팩토링" → "전후 테스트가 모두 통과하는지 확인"

여러 단계 작업은 간략한 계획을 먼저 제시한다.

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

### 5. 한국어 출력 시 문장 끝 콜론 금지

한국어 문장은 마침표로 끝낸다. 콜론으로 끝내지 않는다.

- 다음 줄이 목록이거나 예시여도 문장을 콜론으로 끝내지 않는다.
- 모든 한국어 문장 종결은 `.`, `?`, `!` 중 하나여야 한다.
- 콜론은 코드, 키-값 쌍, 레이블 내부에서만 사용한다. 문장 종결에는 사용하지 않는다.

### 6. 새 파일 첫 줄에 한국어 헤더 주석

새 소스 파일 첫 줄에 해당 파일의 역할을 설명하는 한국어 한 줄 주석을 작성한다.

- TypeScript/JavaScript. `// 사용자 인증 상태를 관리하는 Context Provider`
- Python. `# KIS API 호출을 비동기로 래핑하는 클라이언트`
- SQL. `-- 일별 집계 결과를 저장하는 머티리얼라이즈드 뷰`
- 필수 지시어(`'use client'`, `'use server'`, shebang) 바로 아래에 배치한다.
- 설정 파일(`*.config.ts`, `package.json` 등)은 제외한다.

### 7. 계획 + 체크리스트 + 컨텍스트 노트

비trivial 작업 전에 세 가지 산출물을 먼저 작성한다.

- 계획. 무엇을 왜 만드는지.
- 체크리스트(`checklist.md`). 체크박스 형태의 구체적인 작업 목록. 진행하면서 체크한다.
- 컨텍스트 노트(`context-notes.md`). 작업 중 내린 결정과 그 이유. 계속 추가한다.

### 8. 완료 전 테스트 실행

코드를 수정했다면 완료 선언 전에 테스트를 실행한다.

- `npm test`, `pytest`, `cargo test` 등 프로젝트에 맞는 명령어를 실행한다.
- 테스트가 통과하면 결과를 보고한다. 실패하면 수정 후 재실행한다.
- 테스트 환경이 없으면 최소한 빌드/컴파일이 되는지 확인한다.
- "끝", "완료", "다 됐어" 신호 전에 먼저 실행한다. 요청받은 후가 아니다.

### 9. 시맨틱 커밋

하나의 논리적 변경이 완료되면 커밋한다. 사용자가 요청하기를 기다리지 않는다.

- 기준. "이 커밋을 한 문장으로 설명할 수 있는가." 가능하면 커밋한다. 불가능하면 변경사항이 혼재된 것이다. 분리한다.
- 좋은 예. "auth 미들웨어 추가". 나쁜 예. "auth 추가하고 UI도 고치고 버그도 수정" (3개로 분리).
- 관련 없는 20개의 수정을 쌓아서 개별 롤백 능력을 잃지 않는다.
- 의미 있는 단위에서만 커밋한다.

### 10. 에러를 읽는다. 추측하지 않는다

실제 에러/로그 줄을 읽는다. 기억에서 패턴 매칭하지 않는다.

뭔가 실패했을 때.

- 전체 에러 메시지와 스택 트레이스를 읽는다.
- 예상하는 내용이 아닌 실제 로그 출력을 확인한다.
- 원인을 확인하기 전에 "일반적인 수정"을 적용하지 않는다.
- 불분명하면 print/log를 추가해서 상태를 확인한 후 수정한다.
